# PR Reviewers Assignment Service

Тестовое задание для стажёрской программы Авито (Backend, осень 2025).

REST API сервис на Go + PostgreSQL, который автоматически назначает ревьюеров
на Pull Request'ы, управляет командами и участниками.

**Стек:** Go 1.23 · Gin · PostgreSQL · Clean Architecture · Docker Compose · zap · k6


## Быстрый старт

### Клонировать репозиторий

```shell
    git clone https://github.com/blxxdclxud/PR-reviewers-assigner-avito.git
    cd PR-reviewers-assigner-avito
```

### Настроить переменные окружения

Создать .env из примера

```shell
    cp .env.example .env
```

### Запустить сервис

Создать .env из примера, если нет

```shell
    cp .env.example .env
```

```shell
    docker-compose up --build
```

**ИЛИ**

```shell
    make up
```

### Остановить сервис

```shell
    docker compose down
```

**ИЛИ**

```shell
    make down
```

### Запуск тестов

```shell
    make test # Unit тесты
    make test-integration # Интеграционные тесты
    make test-e2e # E2E тесты
```

#### Сервис будет доступен на `http://localhost:8080`

## Архитектура

Сервис построен по принципам Clean Architecture с тремя явными слоями.
Зависимости направлены только внутрь: внешние слои зависят от внутренних, но не наоборот.
Слой `domain` не зависит ни от чего, кроме стандартной библиотеки.

```
    HTTP Request
         |
         v
+-------------------+
|     Handler       |  internal/adapter/http/handler/
|                   |  - Парсинг JSON-запросов
|  (Gin, DTO <->    |  - Вызов use case через интерфейс
|   domain)         |  - Маппинг domain errors -> HTTP-коды
+--------+----------+
         |  domain objects
         v
+-------------------+
|    Use Case       |  internal/usecase/
|                   |  - Бизнес-правила (кто может быть рецензентом)
|  (транзакции,     |  - Управление транзакциями БД
|   бизнес-логика)  |  - Случайный выбор рецензентов
+--------+----------+
         |  repository interfaces
         v
+-------------------+
|   Repository      |  internal/adapter/postgres/
|                   |  - SQL-запросы
|  (PostgreSQL,     |  - Маппинг sql.ErrNoRows -> domain.ErrNotFound
|   транзакции)     |  - SELECT ... FOR UPDATE для конкурентного доступа
+--------+----------+
         |  SQL / connection pool
         v
+-------------------+
|    PostgreSQL     |
|                   |
|  teams            |
|  users            |
|  pull_requests    |
|  pr_reviewers     |
+-------------------+
```

Интерфейсы репозиториев (`internal/repository/interfaces.go`) и use case
(`internal/adapter/http/handler/`) позволяют тестировать каждый слой изолированно с моками.

## Принятые решения

### 1. Дополнительные коды ошибок

В спецификации описаны доменные ошибки (`TEAM_EXISTS`, `PR_MERGED`, и т.д.). Дополнительно добавлены:

- **`INVALID_INPUT`** (400) — невалидное тело запроса (отсутствуют required поля, неверный JSON)
- **`INTERNAL_ERROR`** (500) — внутренняя ошибка сервера (проблемы с БД, неожиданные ошибки)

**Обоснование:** Для обеспечения полноценности API было необходимо не просто игнорировать эти ошибки, а выводить так же, как и другие задокументированные виды ошибок

### 2. Логирование

Используется structured logging (zap)

### 3. Транзакции

Все операции записи используют транзакции для гарантии консистентности:
- Создание команды с членами
- Создание PR с назначением ревьюверов
- Переназначение ревьюверов

**Обоснование:** Консистентность данных критична.

## Нагрузочное тестирование

Используются две утилиты:
- **[hey](https://github.com/rakyll/hey)** — для идемпотентных и read-only эндпоинтов (одинаковое тело запроса допустимо).
- **[k6](https://k6.io)** — для write-эндпоинтов, где каждый запрос должен содержать уникальные данные (например, уникальный `pull_request_id`). `hey` здесь неприменим: все запросы с одним ID кроме первого вернут 409.

### Подготовка к тестированию

```bash
# Установить k6 (однократно)
curl -L https://github.com/grafana/k6/releases/download/v0.57.0/k6-v0.57.0-linux-amd64.tar.gz \
  | tar -xz -C /tmp && sudo mv /tmp/k6-v0.57.0-linux-amd64/k6 /usr/local/bin/k6

# Запустить сервис
make up

# Создать тестовые данные: 1 команда (15 участников) + 50 PR
make load-seed
```

### Требования к сервису

- Объём данных: до 20 команд и до 200 пользователей
- RPS: 5
- SLI времени ответа: < 300 мс
- SLI успешности: > 99.9%

### Тест 6: POST /pullRequest/reassign — конкурентная замена рецензентов

Проверяет корректность `SELECT ... FOR UPDATE` при параллельных reassign.
Каждый VU работает со своим пулом PR (нет конкуренции за одну строку).
409-ответы (NO_CANDIDATE, NOT_ASSIGNED) — ожидаемое бизнес-поведение, не ошибки.

# или напрямую:
